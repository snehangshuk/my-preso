Why rdev?

Remote development aims to provide:

  • Consistent build environment
  • Predictable fast builds
  • Easy to setup

go/remote-development

- Consistent build environment: As an example, instructors for our bootcamps typically lose a few hours helping new hires set up their laptops for their first build. This will allow most of these problems would go away in a containerized development environment. Remote development environments can provide an effortless way to set up a development environment for new members of a team.

  • Predictable fast builds: Having the builds happen in machines that are close to the services they need to access (e.g. artifactory, multiproduct backend, etc).

  • In rdev, the bare minimum software required are provided in order to do the development. Thus does not conflict with the version installed on the local machine.

More examples on above point; We run code quality tools as part of the build process. For example flake8 and black for Python MPs, ESLint for JS, etc. Having to build in order to get feedback from these tools is not ideal. Most editors and IDEs can be configured to use these tools, but configuring them can be non-trivial so many developers don’t do it. What’s worse, some IDEs like IntelliJ bundle their own code quality tools that might be different than the ones used during the build, which makes errors confusing. Bundling these tools and properly configuring the environment in remote development environments can eliminate this failure mode.

Architecture

Important points to remember:

  • Remote container will be archived and deleted after several days of inactivity, so you cannot use it as persistent storage or a backup host.

  • You can have at most three remote host setups(one for each mp). You can use rdev delete <container name> to delete the ones you don’t want to use anymore.

rdev vs rexec

Remote Execution (rexec) executes the command remotely inside a container, after syncing the state of the local MP with the remote host.

Whereas

Remote Development (rdev) is a container running on a faster host in corp fabric with limited software necessary for development. rexec is a capability added to rdev.

go/rexec
Remote Development(rdev) is a container running on a faster host in corp with limited software necessary for development. Remote Execution(rexec) is a capability added to rdev. In its original form, rdev keeps your source code in a remote host, which you can access via ssh and you can use tools like VS Code with the Remote SSH extension to work completely remotely. For some workflows especially for backend developer and projects that use Java/Python/Scala, it’s more convenient to keep the source code on the local host and execute the commands remotely, after syncing the state of the local MP with the remote host. This is where rdev exec (shortened to rexec) comes into play.

Our IDE team has also integrated rexec with IntelliJ for a seamless experience (go/rexec-demo).
